# Handoff Doc Skill

Spec for writing a developer-facing design handoff doc — either as a Notion page or a local Markdown file in the repo. The user codified this on 2026-05-01 so every feature handoff follows the same structure and PostHog tracking is never forgotten.

## Intake Questions (ALWAYS ask these first, in order)

Before writing anything, ask the user the following. Stop and wait for answers — do not assume defaults beyond what's listed.

1. **Output format** — Notion page or local Markdown file?
   - If the user already pasted a Notion URL, default to Notion and confirm.
   - If they're asking inside the editor without a URL, ask explicitly.

2. **Output location**
   - **Notion:** target page URL (existing empty page) OR a parent page URL under which to create a new page. If a parent, ask for the page title.
   - **Local `.md`:** filepath. Default to `.claude/Feature Requirement Docs/<kebab-case-feature-name>.md` to match the existing project convention. Confirm filename before writing.

3. **Reference handoff** — URL or path of an existing handoff to match in tone/depth (optional but encouraged). Fetch it before writing.

4. **Feature scope** — which screens / sub-features are in scope, and explicitly **what is out of scope** (e.g. sub-flows that exist in the code but should not be documented).

5. **Code area** — the directory or feature folder under `src/` to explore for accurate file paths, component names, and submit/API wiring state.

Only after all five answers are in hand should you proceed to write.

## Format-Specific Syntax

The required sections (below) are identical across Notion and local `.md`, but a few primitives change:

| Element | Notion | Local `.md` |
|---|---|---|
| Table of contents | `<table_of_contents color="gray"/>` | Auto-generated by most readers — omit, or use a manual bulleted list |
| Video placeholder | `<empty-block/>` under a `**Video Recording:**` heading | `> _Video recording: paste Loom link here_` |
| Screenshot placeholder | `<empty-block/>` under a `**Screenshot — <state>:**` heading | `> _Screenshot — <state>: paste image here_` (or `![](path)` once added) |
| Tables | Notion table HTML (`<table header-row="true">...`) | Standard GitHub-flavored Markdown tables |
| Editing existing doc | `update_content` (search-and-replace) preferred; `replace_content` only when empty | Use Edit tool for incremental changes; Write only for new files |
| Footnote / collapsible | `<details><summary>...</summary>...</details>` | Same — GitHub Markdown supports it |

## Required Sections (in order)

1. **Table of Contents** — `<table_of_contents color="gray"/>` at the top.
2. **Feature Summary** — short prose covering:
   - What the feature does (1–3 numbered points)
   - Problem it solves
   - Scope of the handoff (explicit out-of-scope items, if any)
3. **Design Resources** — table with at minimum: Figma File (leave link blank for user), Component Reference (path under `src/`), Route (the entry-point URL).
4. **Implementation Tasks** — one task per screen / sub-feature. Each task MUST contain:
   - **Video Recording:** placeholder `<empty-block/>` (user fills in Loom)
   - **Description** — prose, ending with a **Key Features** bulleted list
   - **Screenshot — &lt;state name&gt;:** one or more `<empty-block/>` placeholders
   - **### Subtasks** — actionable checklist (`- [ ]`) prefixed with `UI Change:`, `Logic Change:`, or `Data Fetch:`
   - **### Components Involved** — table: Component | Type (Existing / New / Modified) | Reference (file path)
   - **### Backend Services** — table: Service | Endpoint | Status. Use `[To be added from production backend]` if unknown.
   - **### PostHog Tracking** — table: Event Name | Trigger | Properties | Status. See "PostHog Conventions" below.
5. **Open Items / Notes for Developer** — known no-ops, follow-ups, scope exclusions.
6. **Acceptance Criteria** — split into one block per task, plus a `**PostHog Tracking:**` block, plus a `**General:**` block. All items are `- [ ]`.
7. **Footer line** — italicized: `*Design handoff for <feature name>. Last updated: <Month Year>*`

## PostHog Conventions

**Defer to the `posthog` skill at `.claude/skills/posthog/SKILL.md` for full standards.** Load that skill before filling the PostHog Tracking section of any handoff. The skill defines the event taxonomy, property rules, funnel design, identification, and anti-patterns — they should not be duplicated here.

Quick summary (for reference, not as a substitute for loading the skill):

- Event names use `[object]_[verb]` snake_case (e.g. `policy_selected`, `eval_run_submitted`)
- Each task MUST include all four event categories: entry, key action(s), success/completion, exit/error
- Properties must include IDs, counts, and categorical context — never PII, secrets, or free-text
- Map each task's events to a numbered funnel; the tracking table should include a `Funnel Step` column when events form a funnel
- Tie events to person and company via `posthog.identify()` and `posthog.group('company', ...)`
- Status column starts at `To be added`, flips to `Implemented` once wired up

The PostHog Tracking table schema in handoffs is:

| Funnel Step | Event Name | Trigger | Properties | Status |
|---|---|---|---|---|

Drop the `Funnel Step` column only when the events are ambient (don't form a single funnel).

## UI Guideline Reminders (per project CLAUDE.md)

When writing subtasks and acceptance criteria, always include reminders that the implementation must:
- Use only Tailwind colors `gray`, `red`, `green`, `amber`
- Use `bg-gray-0` instead of `bg-white`
- Use Title Case for headings, titles, and labels

## Workflow

1. **Run the Intake Questions** above — get all five answers before doing anything else.
2. **Fetch a reference doc** if the user provided one — match its tone, depth, and table structure.
3. **Explore the code** to get accurate file paths, component names, and the current state of submit/API wiring. Use a sub-agent for this (Explore subagent) when the feature spans more than a couple of files.
4. **Confirm scope** back to the user one more time before writing — especially what is *excluded*.
5. **Write to the chosen destination** using the format-specific syntax table above. Leave placeholders, not guesses — every video and screenshot is a placeholder for the user to fill in manually.
6. **Flag unknowns explicitly** in the Open Items section rather than inventing details.

## Anti-Patterns to Avoid

- Don't fabricate Figma URLs, Loom URLs, or backend endpoints — always leave them blank or marked `[To be added from production backend]`.
- Don't include implementation code in the handoff body unless the reference doc does (use a `<details>` block if so).
- Don't claim a feature is "complete" or "shipped" if the code shows the submit handler is still a no-op — call it out in Open Items.
- Don't expand scope beyond what the user asked for. If they say "just these two screens," do not document peripheral sub-flows even if they exist in the code.
